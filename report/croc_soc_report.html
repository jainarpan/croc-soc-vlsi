1. Introduction & Objectives

This report documents the experimental learning assignment on the Croc System-on-Chip (SoC), an open-source, education-focused chip designed by the PULP Platform team at ETH Zurich and the University of Bologna. The Croc SoC was successfully taped out in November 2024 as the chip MLEM in IHP's open-source 130 nm technology, and its core functionality was verified on real silicon in early 2026.

The assignment covers the complete ASIC design flow — from RTL study and simulation to logic synthesis and physical design — using freely available open-source EDA tools. This is in line with the growing ecosystem of open-source silicon, enabled by Google-sponsored PDK releases and community tools such as Yosys, OpenROAD, and KLayout.

1.1 Objectives

  1. Understand the Git version control system and its role in hardware design projects.
  2. Study the Croc SoC architecture — processor, interconnect, memory, and peripherals.
  3. Identify and understand the free/open-source EDA tool-chain available for smaller designs.
  4. Download the RTL source and tool configuration from GitHub.
  5. Compile the SystemVerilog code using Verilator inside a Docker environment.
  6. Develop a testbench with multiple register Read/Write (RD/WR) tests.
  7. Use Yosys to synthesise the RTL to the IHP 130 nm technology library.
  8. Use OpenROAD & KLayout for floorplan, place-and-route, and GDS layout.

1.2 About PULP Platform

The Parallel Ultra-Low-Power (PULP) platform is a joint research initiative between ETH Zurich and the University of Bologna. It focuses on energy-efficient multi-core platforms for IoT, edge AI, and embedded applications. PULP has produced dozens of open-source processor cores (Ibex/CVE2, CV32E40P, Snitch), interconnects (OBI, AXI), and complete SoCs (Cheshire, Croc). All IP is published under permissive open-source licences (Solderpad, Apache 2.0).

2. GIT Version Control System

Git is a distributed version control system (DVCS) created by Linus Torvalds in 2005. In hardware design, Git is used to track RTL changes, manage IP dependencies, collaborate across teams, and maintain a reproducible design history — critical for silicon tape-outs where any RTL change can have costly consequences.

2.3 Bender — RTL Dependency Manager

Croc uses Bender, a dependency manager for hardware projects (analogous to npm for JavaScript or cargo for Rust). Each IP block has a Bender.yml describing its source files and dependencies. The top-level Bender.lock pins exact git commits for all dependencies, ensuring bit-for-bit reproducibility across machines and time. 

# Key Bender commands used in this project
bender checkout          # Download all deps at pinned commits into .bender/
bender script verilator  # Generate file list for Verilator
bender script synth      # Generate file list for Yosys synthesis

3.3 OBI Interconnect

Croc uses the Open Bus Interface (OBI), a simple request/response bus protocol maintained by the OpenHW Group. OBI was chosen over AXI for its simplicity — it has fewer signals, no separate address and data channels to manage, and is easy to understand in RTL.

5.3 SystemVerilog Compilation with Verilator

Verilator compiles SystemVerilog into a cycle-accurate C++ model. The compilation step also performs thorough lint checking — catching undeclared signals, undriven outputs, width mismatches, and other common RTL errors. 

# Step 1: Generate file list with Bender
bender script verilator -t verilator > croc_flist.f

# Step 2: Lint-only check
verilator --lint-only -sv \
  -f croc_flist.f \
  --top-module croc_soc

# Step 3: Build simulation binary with our testbench
verilator --binary -sv \
  -f croc_flist.f \
  testbench/croc_tb.sv \
  --top-module croc_tb \
  --trace \
  -Mdir results/simulation/obj_dir

# Step 4: Run simulation
./results/simulation/obj_dir/Vcroc_tb

The -t verilator Bender target selects technology-independent implementations (generic SRAM, generic clock cells) rather than the IHP-specific macros — ensuring the simulation compiles without PDK dependencies.

6.3 Key Testbench Code Excerpt

// OBI Write Task
task automatic obi_write(input logic [31:0] addr, data, input logic [3:0] be);
  @(posedge clk);
  obi_req <= 1; obi_we <= 1; obi_addr <= addr;
  obi_wdata <= data; obi_be <= be;
  wait (obi_gnt == 1);          // wait for crossbar grant
  @(posedge clk); obi_req <= 0;
  wait (obi_rvalid == 1);       // wait for write response
  @(posedge clk);
endtask

// OBI Read + Verify Task
task automatic obi_read_check(input logic [31:0] addr, expected, mask);
  @(posedge clk);
  obi_req <= 1; obi_we <= 0; obi_addr <= addr; obi_be <= 4'hF;
  wait (obi_gnt == 1);
  @(posedge clk); obi_req <= 0;
  wait (obi_rvalid == 1);
  // Compare with mask — allows partial field checking
  if ((obi_rdata & mask) !== (expected & mask)) tests_failed++;
  else tests_passed++;
endtask

7. Logic Synthesis with Yosys

7.1 Synthesis Flow

Yosys is an open-source RTL synthesis tool. For SystemVerilog support, it uses the yosys-slang frontend plugin, which handles complex SV features such as parameterised structs, packed arrays, and interfaces. The synthesis targets the IHP SG13G2 standard cell library at the typical-typical (TT) corner. 

# Navigate to synthesis directory (inside Docker container)
cd /fosic/designs/croc/yosys

# Generate synthesis file list
bender script flist -t synthesis -t ihp13 > syn_flist.f

# Run synthesis
./run_synthesis.sh --synth

# Key Yosys script steps (run_synthesis.tcl):
# 1. plugin -i slang      → load SV parser
# 2. read_slang -f flist  → parse all .sv files
# 3. hierarchy -check     → verify module hierarchy
# 4. synth -top croc_soc  → elaborate + optimise
# 5. dfflibmap            → map flip-flops to library
# 6. abc -liberty         → technology map combinational logic
# 7. write_verilog        → output gate-level netlist

8. Floorplan & Physical Design (OpenROAD + KLayout)

8.1 OpenROAD Flow

OpenROAD provides a fully automated RTL-to-GDS backend. It takes the gate-level netlist from Yosys, along with placement constraints and the IHP technology LEF/LIB files, and performs the complete physical design flow through to a routed layout. 

# Run complete backend flow (inside Docker, from croc/openroad/)

9.4 Key Learnings

  1. Open-source ASIC is real and production-proven — the MLEM chip from this exact RTL was verified on silicon in 2026, proving the entire open-source flow (Yosys → OpenROAD → KLayout → IHP 130nm) produces manufacturable chips.
  2. OBI is simpler than AXI for small, in-order SoCs — the minimal signal set (req/gnt/rvalid/rdata) makes it easy to write testbenches and understand bus timing.
  3. Bender solves a real problem — managing 170 RTL files across 15+ IP blocks without a dependency manager would be extremely fragile. Bender's lock file ensures reproducibility.
  4. CVE2 is an extremely compact RISC-V core — 2 pipeline stages, no cache, no MMU, ~3000 flip-flops, yet fully functional for bare-metal software.
  5. Docker removes all environment friction — the single command start_vnc.bat provides a complete, ready-to-use Linux EDA environment on Windows.

10. Conclusion

This assignment successfully completed the full study and partial implementation of the Croc System-on-Chip developed by the PULP Platform at ETH Zurich. The work covered all six required tasks: repository management with Git, architecture study, RTL compilation, testbench development, logic synthesis, and physical design.

The Croc SoC represents an outstanding example of modern open-source chip design. By combining a production-verified RISC-V processor core (CVE2), a clean OBI interconnect, essential peripherals, and a completely open EDA flow targeting a real 130 nm process (IHP SG13G2), it provides students with a genuine taste of industrial ASIC design — at zero cost.

The open-source EDA ecosystem — Yosys for synthesis, OpenROAD for place and route, and KLayout for GDS — has matured to the point where real chips can be taped out. The MLEM chip (Croc v1 tapeout, November 2024) demonstrates this concretely: silicon verified in 2026.

The testbench developed in this assignment (croc_tb.sv) exercises the OBI bus with 9 test cases covering SRAM read/write, byte-enable testing, GPIO register access, timer configuration, and UART peripheral writes — providing a solid verification foundation that could be extended into a more complete verification environment (UVM or directed random tests).

All work is publicly available on GitHub at https://github.com/jainarpan/croc-soc-vlsi, with commit history demonstrating the incremental development process as expected in a professional VLSI design workflow.
